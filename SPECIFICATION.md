## 1. Goals and Non-Goals

### Goals

1. **API Compatibility**: Implement the Jira REST API v2 endpoints consumed by `src/assistant_mcp/clients/jira_client.py` and `src/assistant_mcp/tools/jira/` — the client code must work against the emulator with zero modifications.
2. **Single Container**: Run from a single container image with no external dependencies (no PostgreSQL, no Redis, no Elasticsearch).
3. **SQLite Backend**: Use SQLite as the sole persistence layer, keeping operational simplicity.
4. **JSON Import**: Support bulk import of Jira issue JSON files (matching the export format in `mcp_session_data/jira_sessions/`) to seed the emulator with real ticket data.
5. **RHOAI Workflow Support**: Model the specific Jira projects, issue types, workflows, and custom fields used in the RHOAI development lifecycle (RHAIRFE, RHAISTRAT, RHOAIENG, AIPCC).
6. **JQL Support**: Implement a JQL parser and query engine sufficient for the queries generated by the client code.
7. **Minimal Web UI**: Provide a basic read-only web interface for browsing issues (not a full Jira Cloud UI clone).
8. **Deterministic Testing**: Enable reproducible test scenarios for agent-driven SDLC workflows.

### Non-Goals

- Full Jira Cloud feature parity (boards, sprints management, dashboards, permissions schemes, workflows designer, etc.)
- Atlassian Document Format (ADF) rendering — store as plain text
- Attachments / file upload
- Webhooks
- OAuth / Atlassian Connect
- Email notifications
- Agile board management (Scrum/Kanban boards)
- Time tracking
- Advanced permission schemes (project roles, issue security levels)
- Confluence/Bitbucket integration
- Jira Service Management features (SLA, queues, approvals)

### Request Flow

1. HTTP request arrives at FastAPI router
2. Router dispatches to appropriate handler
3. Handler calls service layer
4. Service layer validates input, applies business logic (workflow transitions, field validation)
5. Service layer interacts with SQLite via SQLAlchemy ORM
6. For search: JQL string is parsed into an AST, then translated to SQL WHERE clauses
7. Response is formatted to match Jira REST API v2 JSON structure

## 3. Technology Stack

| Component | Technology | Rationale |
|-----------|-----------|-----------|
| Language | Python 3.11+ | Matches the client codebase, shared ecosystem |
| Web Framework | FastAPI | Async support, automatic OpenAPI docs, Pydantic validation |
| Database | SQLite (via aiosqlite) | Zero-config, single-file, sufficient for emulator scale |
| ORM | SQLAlchemy 2.0 (async) | Mature, well-documented, handles schema migrations |
| JQL Parser | lark-parser | EBNF grammar support, generates clean ASTs, well-maintained |
| Web UI Templates | Jinja2 | Built into FastAPI/Starlette, simple server-side rendering |
| CSS Framework | Pico CSS or Simple.css | Minimal, classless CSS for basic UI without build tools |
| Container | Podman / Docker | Standard OCI image |
| Base Image | `python:3.11-slim` | Small footprint |
| Migration | Alembic | SQLAlchemy migration tool (optional for initial release) |
| Testing | pytest + httpx | Async test client for FastAPI |

## 4. Data Model

### 4.1 SQLite Schema

#### `projects`

```sql
CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    key TEXT UNIQUE NOT NULL,          -- e.g., "RHAIRFE", "RHAISTRAT"
    name TEXT NOT NULL,                -- e.g., "Red Hat AI RFE project"
    description TEXT,
    lead TEXT,                         -- username of project lead
    project_type_key TEXT DEFAULT 'software',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### `issue_types`

```sql
CREATE TABLE issue_types (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,         -- "Bug", "Task", "Story", "Feature", "Feature Request", "Initiative", "Epic", "Sub-task"
    description TEXT,
    subtask BOOLEAN DEFAULT FALSE,
    icon_url TEXT
);
```

#### `project_issue_types`

```sql
CREATE TABLE project_issue_types (
    project_id INTEGER REFERENCES projects(id),
    issue_type_id INTEGER REFERENCES issue_types(id),
    PRIMARY KEY (project_id, issue_type_id)
);
```

#### `priorities`

```sql
CREATE TABLE priorities (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,         -- "Blocker", "Critical", "Major", "Normal", "Minor", "Undefined"
    icon_url TEXT,
    sort_order INTEGER DEFAULT 0
);
```

#### `statuses`

```sql
CREATE TABLE statuses (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,         -- "New", "Open", "In Progress", "Review", "Closed", etc.
    description TEXT,
    category TEXT DEFAULT 'new'        -- "new", "indeterminate", "done" (status categories)
);
```

#### `resolutions`

```sql
CREATE TABLE resolutions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,         -- "Done", "Won't Do", "Duplicate", "Cannot Reproduce"
    description TEXT
);
```

#### `workflows`

```sql
CREATE TABLE workflows (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT UNIQUE NOT NULL,         -- "RHAIRFE Workflow", "Default", "RHAISTRAT Workflow"
    description TEXT
);
```

#### `workflow_transitions`

```sql
CREATE TABLE workflow_transitions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    workflow_id INTEGER REFERENCES workflows(id),
    name TEXT NOT NULL,                -- "Start Progress", "Approve", "Close"
    from_status_id INTEGER REFERENCES statuses(id),  -- NULL = global transition (from any)
    to_status_id INTEGER REFERENCES statuses(id) NOT NULL
);
```

#### `project_workflows`

```sql
CREATE TABLE project_workflows (
    project_id INTEGER REFERENCES projects(id),
    issue_type_id INTEGER REFERENCES issue_types(id),  -- NULL = default for project
    workflow_id INTEGER REFERENCES workflows(id),
    PRIMARY KEY (project_id, COALESCE(issue_type_id, 0))
);
```

#### `users`

```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT UNIQUE NOT NULL,      -- login name / account ID
    display_name TEXT NOT NULL,
    email TEXT,
    active BOOLEAN DEFAULT TRUE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

#### `issues`

```sql
CREATE TABLE issues (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    key TEXT UNIQUE NOT NULL,           -- "RHAIRFE-123" (generated from project key + sequence)
    project_id INTEGER REFERENCES projects(id) NOT NULL,
    issue_type_id INTEGER REFERENCES issue_types(id) NOT NULL,
    summary TEXT NOT NULL,
    description TEXT,
    status_id INTEGER REFERENCES statuses(id) NOT NULL,
    priority_id INTEGER REFERENCES priorities(id),
    resolution_id INTEGER REFERENCES resolutions(id),
    assignee_id INTEGER REFERENCES users(id),
    reporter_id INTEGER REFERENCES users(id),
    parent_id INTEGER REFERENCES issues(id),  -- for subtasks and epic children
    due_date DATE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    resolved_at TIMESTAMP
);
```

#### `issue_sequences`

```sql
CREATE TABLE issue_sequences (
    project_id INTEGER PRIMARY KEY REFERENCES projects(id),
    next_number INTEGER DEFAULT 1
);
```

#### `labels`

```sql
CREATE TABLE labels (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    issue_id INTEGER REFERENCES issues(id) ON DELETE CASCADE,
    label TEXT NOT NULL,
    UNIQUE(issue_id, label)
);
```

#### `components`

```sql
CREATE TABLE components (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER REFERENCES projects(id),
    name TEXT NOT NULL,
    description TEXT,
    lead TEXT,
    UNIQUE(project_id, name)
);
```

#### `issue_components`

```sql
CREATE TABLE issue_components (
    issue_id INTEGER REFERENCES issues(id) ON DELETE CASCADE,
    component_id INTEGER REFERENCES components(id),
    PRIMARY KEY (issue_id, component_id)
);
```

#### `versions`

```sql
CREATE TABLE versions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER REFERENCES projects(id),
    name TEXT NOT NULL,                 -- "rhoai-3.4.EA1", "rhoai-3.4.EA2"
    description TEXT,
    released BOOLEAN DEFAULT FALSE,
    release_date DATE,
    start_date DATE,
    sort_order INTEGER DEFAULT 0,
    UNIQUE(project_id, name)
);
```

#### `issue_fix_versions`

```sql
CREATE TABLE issue_fix_versions (
    issue_id INTEGER REFERENCES issues(id) ON DELETE CASCADE,
    version_id INTEGER REFERENCES versions(id),
    PRIMARY KEY (issue_id, version_id)
);
```

#### `issue_affects_versions`

```sql
CREATE TABLE issue_affects_versions (
    issue_id INTEGER REFERENCES issues(id) ON DELETE CASCADE,
    version_id INTEGER REFERENCES versions(id),
    PRIMARY KEY (issue_id, version_id)

```

#### `comments`

```sql
CREATE TABLE comments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
```

```sql
CREATE TABLE custom_fields (
```

## 5. REST API Specification

All endpoints are under `/rest/api/2/`. The emulator returns JSON with `Content-Type: application/json`.

### 5.1 Authentication Endpoint

#### `GET /rest/api/2/myself`

Returns the current user. Used by the client to validate authentication.

**Response 200:**
```json
{
  "self": "http://localhost:8080/rest/api/2/myself",
  "accountId": "admin",
  "displayName": "Admin User",
  "emailAddress": "admin@example.com",
  "active": true
}
```

**Authentication Handling:**

The emulator supports two authentication modes to match the client:
- **Basic Auth**: `Authorization: Basic base64(email:api_token)` — for Jira Cloud compatibility
- **Bearer Token**: `Authorization: Bearer {token}` — for Jira Server/DC compatibility

In the default (permissive) mode, any valid auth header is accepted. An optional strict mode can validate against the `users` table.

### 5.2 Issue Endpoints

#### `POST /rest/api/2/issue`

Create a new issue.

**Request:**
```json
{
  "fields": {
    "project": {"key": "RHAIRFE"},
    "summary": "New feature request",
    "description": "Description text",
    "issuetype": {"name": "Feature Request"},
    "priority": {"name": "Major"},
    "assignee": {"name": "jdoe"},
    "labels": ["label1", "label2"],
    "components": [{"name": "Dashboard"}],
    "customfield_12310243": 5
  }
}
```

**Response 201:**
```json
{
  "id": "10001",
  "key": "RHAIRFE-42",
  "self": "http://localhost:8080/rest/api/2/issue/10001"
}
```

**Behavior:**
- Auto-generate issue key from project key + next sequence number
- Validate project exists
- Validate issue type is allowed for project
- Set status to the first status in the project's workflow
- Create/find labels, components as needed
- Look up or auto-create users referenced by `assignee`/`reporter`
- Store custom field values

#### `PUT /rest/api/2/issue/{issueIdOrKey}`

Update an issue. Supports both `fields` and `update` styles.

**Request (fields style):**
```json
{
  "fields": {
    "summary": "Updated summary",
    "priority": {"name": "Critical"},
    "assignee": {"name": "jsmith"}
  }
}
```

**Request (update style):**
```json
{
  "update": {
    "labels": [
      {"add": "new-label"},
      {"remove": "old-label"}
    ],
    "components": [
      {"add": {"name": "Engine"}}
    ],
    "comment": [
      {"add": {"body": "A new comment"}}
    ]
  }
}
```

**Response:** 204 No Content

**Behavior:**
- Update `updated_at` timestamp
- For `update` operations: support `set`, `add`, `remove` verbs on array fields
- For `fields`: direct field replacement
- Mixed `fields` + `update` in one request: each field appears in only one

#### `DELETE /rest/api/2/issue/{issueIdOrKey}`

Delete an issue and all associated data (comments, links, labels, watchers, custom field values).

**Response:** 204 No Content

### 5.3 Transition Endpoints

#### `GET /rest/api/2/issue/{issueIdOrKey}/transitions`

Get available transitions for the issue's current status.

**Query Parameters:**
- `expand` — if includes `transitions.fields`, include required fields for each transition

**Response 200:**
```json
{
  "expand": "transitions",
  "transitions": [
    {
      "id": "5",
      "name": "Start Progress",
      "to": {
        "self": "http://localhost:8080/rest/api/2/status/3",
        "id": "3",
        "name": "In Progress",
        "statusCategory": {
          "id": 2,
          "key": "indeterminate",
          "name": "In Progress"
        }
      }
    },
    {
      "id": "10",
      "name": "Close",
      "to": {
        "self": "http://localhost:8080/rest/api/2/status/6",
        "id": "6",
        "name": "Closed",
        "statusCategory": {
          "id": 3,
          "key": "done",
          "name": "Done"
        }
      }
    }
  ]
}
```

#### `POST /rest/api/2/issue/{issueIdOrKey}/transitions`

Perform a status transition.

**Request:**
```json
{
  "transition": {"id": "5"}
}
```

**Response:** 204 No Content

**Behavior:**
- Validate that the requested transition is available from the issue's current status
- Update the issue's status
- If transition targets a "done" category status and no resolution set, auto-set resolution to "Done"
- Update `updated_at` and optionally `resolved_at`

### 5.4 Comment Endpoints

#### `GET /rest/api/2/issue/{issueIdOrKey}/comment`

Get all comments for an issue.

**Response 200:**
```json
{
  "startAt": 0,
  "maxResults": 50,
  "total": 2,
  "comments": [
```

## 6. JQL Engine

### 6.1 Grammar (EBNF for lark-parser)

```ebnf
start: query

query: clause (KEYWORD clause)* orderby?

clause: "(" query ")"
      | NOT clause
      | field operator value

KEYWORD: AND | OR

NOT: "NOT" | "not" | "!"

operator: "=" | "!=" | ">" | ">=" | "<" | "<="
        | "~" | "!~"
        | IN | NOT_IN
        | IS | IS_NOT

IN: "IN" | "in"
NOT_IN: "NOT" "IN" | "not" "in"
IS: "IS" | "is"
IS_NOT: "IS" "NOT" | "is" "not"

AND: "AND" | "and" | "&&"
OR: "OR" | "or" | "||"

field: FIELD_NAME
     | QUOTED_FIELD

value: STRING
     | QUOTED_STRING
     | NUMBER
     | function_call
     | EMPTY
     | NULL
     | value_list

value_list: "(" value ("," value)* ")"

function_call: FUNCTION_NAME "(" (value ("," value)*)? ")"

FUNCTION_NAME: /[a-zA-Z_]\w*/
FIELD_NAME: /[a-zA-Z_][\w.]*/

orderby: ORDER_BY field (ASC | DESC)? ("," field (ASC | DESC)?)*
ORDER_BY: "ORDER" "BY" | "order" "by"
ASC: "ASC" | "asc"
DESC: "DESC" | "desc"

EMPTY: "EMPTY" | "empty"
NULL: "NULL" | "null"
```

### 6.2 JQL to SQL Translation

The JQL engine translates parsed AST nodes into SQLAlchemy filter expressions:

| JQL | SQL Equivalent |
|-----|---------------|
| `project = RHAIRFE` | `WHERE issues.project_id = (SELECT id FROM projects WHERE key = 'RHAIRFE')` |
| `status = "In Progress"` | `WHERE issues.status_id = (SELECT id FROM statuses WHERE name = 'In Progress')` |
| `assignee = jdoe` | `WHERE issues.assignee_id = (SELECT id FROM users WHERE username = 'jdoe')` |
| `summary ~ "architecture"` | `WHERE issues.summary LIKE '%architecture%'` (case-insensitive via COLLATE NOCASE) |
| `text ~ "keyword"` | `WHERE (issues.summary LIKE '%keyword%' OR issues.description LIKE '%keyword%')` |
| `labels = "urgent"` | `WHERE EXISTS (SELECT 1 FROM labels WHERE labels.issue_id = issues.id AND labels.label = 'urgent')` |
| `labels IN ("a", "b")` | `WHERE EXISTS (SELECT 1 FROM labels WHERE labels.issue_id = issues.id AND labels.label IN ('a', 'b'))` |
| `priority IN (Blocker, Critical)` | `WHERE issues.priority_id IN (SELECT id FROM priorities WHERE name IN ('Blocker', 'Critical'))` |
| `created >= "2026-01-01"` | `WHERE issues.created_at >= '2026-01-01'` |
| `assignee IS EMPTY` | `WHERE issues.assignee_id IS NULL` |
| `resolution IS NOT EMPTY` | `WHERE issues.resolution_id IS NOT NULL` |
| `key = RHAIRFE-42` | `WHERE issues.key = 'RHAIRFE-42'` |
| `issuetype = Bug` | `WHERE issues.issue_type_id = (SELECT id FROM issue_types WHERE name = 'Bug')` |
| `component = Dashboard` | `WHERE EXISTS (SELECT 1 FROM issue_components ic JOIN components c ON ic.component_id = c.id WHERE ic.issue_id = issues.id AND c.name = 'Dashboard')` |
| `fixVersion = "rhoai-3.4"` | `WHERE EXISTS (SELECT 1 FROM issue_fix_versions ifv JOIN versions v ON ifv.version_id = v.id WHERE ifv.issue_id = issues.id AND v.name = 'rhoai-3.4')` |
| `currentUser()` | Resolved to the authenticated user's username |

### 6.3 Supported Fields

**Direct column fields:**
- `project` → projects.key
- `issuetype` / `type` → issue_types.name
- `status` → statuses.name
- `priority` → priorities.name
- `assignee` → users.username (via assignee_id)
- `reporter` → users.username (via reporter_id)
- `summary` → issues.summary
- `description` → issues.description
- `resolution` → resolutions.name
- `created` → issues.created_at
- `updated` → issues.updated_at
- `due` / `duedate` → issues.due_date
- `key` / `issuekey` → issues.key
- `parent` → parent issue key

**Join fields (EXISTS subqueries):**
- `labels` → labels table
- `component` → issue_components + components tables
- `fixVersion` → issue_fix_versions + versions tables
- `affectedVersion` → issue_affects_versions + versions tables

**Text search (CONTAINS operator `~`):**
- `text` → summary + description (combined search)
- `summary` → summary only
- `description` → description only
- `comment` → comments.body

**Custom fields:**
- `cf[NNNNN]` → issue_custom_field_values table

### 6.4 Functions

| Function | Implementation |
|----------|---------------|
| `currentUser()` | Returns authenticated user's username |
| `now()` | Returns current UTC timestamp |
| `startOfDay()` | Returns start of current day (00:00:00 UTC) |
| `endOfDay()` | Returns end of current day (23:59:59 UTC) |
| `startOfWeek()` | Returns Monday 00:00:00 of current week |
| `endOfWeek()` | Returns Sunday 23:59:59 of current week |
| `startOfMonth()` | Returns 1st of current month 00:00:00 |
| `endOfMonth()` | Returns last day of current month 23:59:59 |
| `startOfYear()` | Returns Jan 1 00:00:00 of current year |
| `endOfYear()` | Returns Dec 31 23:59:59 of current year |
| `EMPTY` / `NULL` | Translates to SQL `IS NULL` check |

Increment syntax (`-1d`, `2w`, etc.) is supported for time functions.

## 7. Workflow Engine

### 7.1 RHAIRFE Workflow

```
New → Stakeholder Review → Review → Pending Approval → Approved
                                  ↘ Rejection Pending → Closed
```

| From Status | Transition Name | To Status |
|-------------|----------------|-----------|
| New | Submit for Review | Stakeholder Review |
| Stakeholder Review | Begin Review | Review |
| Review | Accept | Pending Approval |
| Review | Reject | Rejection Pending |
| Pending Approval | Approve | Approved |
| Pending Approval | Reopen | Review |
| Rejection Pending | Close | Closed |
| Rejection Pending | Reopen | Review |
| Any | Close | Closed |

### 7.2 RHAISTRAT Workflow

```
New → Refinement → In Progress → Review → Done
                                        ↘ Closed
```

| From Status | Transition Name | To Status |
|-------------|----------------|-----------|
| New | Start Refinement | Refinement |
| Refinement | Start Progress | In Progress |
| In Progress | Submit for Review | Review |
| Review | Complete | Done |
| Review | Reopen | In Progress |
| Any | Close | Closed |

### 7.3 Default Workflow (RHOAIENG, AIPCC, and others)

```
New / Backlog / To Do → In Progress → Code Review → Testing → Done
                                                            ↘ Closed
```

| From Status | Transition Name | To Status |
|-------------|----------------|-----------|
| New | Start | To Do |
| To Do | Start Progress | In Progress |
| Backlog | Start Progress | In Progress |
| In Progress | Submit for Review | Code Review |
| Code Review | Start Testing | Testing |
| Code Review | Reopen | In Progress |
| Testing | Complete | Done |
| Testing | Reopen | In Progress |
| Any | Close | Closed |

### 7.4 Workflow Resolution

When determining which workflow to use for an issue:
1. Look up `project_workflows` for the issue's project + issue type
2. If not found, look up `project_workflows` for the project with `issue_type_id = NULL` (default)
3. If not found, use the "Default" workflow

### 7.5 Transition Behavior

- When transitioning to a "done" category status: auto-set resolution to "Done" if not already set, set `resolved_at`
- When transitioning FROM a "done" category status: clear resolution and `resolved_at`
- Always update `updated_at` on transition

## 8. Authentication

### 8.1 Default Mode (Permissive)

- Accept any `Authorization` header (Basic or Bearer)
- Extract username from Basic auth email or create a default user
- For Bearer tokens, use a configurable default user
- All API calls succeed regardless of token validity
- The `/rest/api/2/myself` endpoint returns the extracted user

### 8.2 Strict Mode (Optional)

When `AUTH_MODE=strict` is set:
- Validate Basic auth credentials against `users` table
- Validate Bearer tokens against a `tokens` table
- Return 401 for invalid credentials

### 8.3 No-Auth Mode

When `AUTH_MODE=none`:
- No Authorization header required
- All requests use a default "anonymous" user

### 9.1 Import Format

The emulator accepts JSON files matching the format exported by the `assistant_mcp` Jira session system. Each file represents one issue:

```json

  "id": "5463484",

  "description": "Description text...",
  "status": "New",
  "priority": "Undefined",
  "issue_type": "Feature Request",
  "assignee": "Erwan Gallen",
  "reporter": "Prarit Bhargava",
  "created": "2025-05-15T16:00:08.196+0000",
  "updated": "2025-11-12T15:43:34.263+0000",
  "project": {
    "key": "RHAIRFE",
    "name": "Red Hat AI RFE project"
  },
  "components": [],
  "labels": ["Gaudi", "Intel"],
  "affects_versions": [],
  "fix_versions": [],
  "target_versions": [],
  "resolution": null,
  "due_date": null,
  "team": null,
  "target_start": null,
  "target_end": null,
  "epic_link": null,
  "story_points": null,
  "sprints": [],
  "affects_testing": [],
  "release_blocker": null,
  "severity": null
}
```

### 9.2 Import Behavior

1. **Auto-create projects**: If a project referenced by an issue doesn't exist, create it from the `project.key` and `project.name` fields
2. **Auto-create users**: If an assignee/reporter doesn't exist, create a user with `display_name` as the provided name and a slugified username
3. **Auto-create issue types**: If the `issue_type` doesn't exist, create it
4. **Auto-create statuses**: If the `status` doesn't exist, create it with "indeterminate" category
5. **Auto-create priorities**: If the `priority` doesn't exist, create it
6. **Auto-create components**: Create any referenced components
7. **Auto-create versions**: Create any referenced versions (fix_versions, affects_versions, target_versions)
8. **Issue key preservation**: Use the original issue key from the JSON (don't auto-generate)
9. **Sequence tracking**: After import, update `issue_sequences` to be max(imported key number) + 1
10. **Custom field mapping**: Map the assistant_mcp export fields to custom fields:
    - `team` → `customfield_12313240`
    - `story_points` → `customfield_12310243`
    - `target_start` → `customfield_12313941`
    - `target_end` → `customfield_12313942`
    - `affects_testing` → `customfield_12310170`
    - `release_blocker` → `customfield_12319743`
    - `severity` → `customfield_12316142`
11. **Epic links**: If `epic_link` is set, create a parent-child relationship
12. **Sprints**: Create sprint records and associate issues
13. **Idempotency**: If an issue with the same key already exists, update it rather than failing

| Export Field | Emulator Target | Notes |
|-------------|-----------------|-------|
| `key` | `issues.key` | Preserved as-is |
| `id` | (ignored) | Emulator generates its own IDs |
| `summary` | `issues.summary` | Direct |
| `description` | `issues.description` | Direct |
| `status` | `issues.status_id` → `statuses.name` | Lookup or auto-create |
| `priority` | `issues.priority_id` → `priorities.name` | Lookup or auto-create |
| `issue_type` | `issues.issue_type_id` → `issue_types.name` | Lookup or auto-create |
| `assignee` | `issues.assignee_id` → `users.display_name` | Lookup or auto-create user |
| `reporter` | `issues.reporter_id` → `users.display_name` | Lookup or auto-create user |
| `created` | `issues.created_at` | Parse ISO 8601 |
| `updated` | `issues.updated_at` | Parse ISO 8601 |
| `project.key` | `issues.project_id` → `projects.key` | Lookup or auto-create |
| `project.name` | `projects.name` | Used when auto-creating |
| `components[]` | `issue_components` | Auto-create components |
| `labels[]` | `labels` | Direct insert |

| `resolution` | `issues.resolution_id` → `resolutions.name` | Lookup or auto-create |
| `due_date` | `issues.due_date` | Direct |
| `team` | `customfield_12313240` | Custom field value |
| `story_points` | `customfield_12310243` | Custom field value |
| `target_start` | `customfield_12313941` | Custom field value |
| `target_end` | `customfield_12313942` | Custom field value |

| `sprints[]` | `issue_sprints` | Auto-create sprints |
| `affects_testing[]` | `customfield_12310170` | Custom field value (JSON) |
| `release_blocker` | `customfield_12319743` | Custom field value |
| `severity` | `customfield_12316142` | Custom field value |
